Abstract & Contents

Abstract

This case study investigates the algorithmic design, implementation, and theoretical complexity of three distinct graph-search and sorting strategies applied to the Inertia Puzzle Game. The Inertia game presents a player on a grid who slides in eight directions, collecting gems while avoiding mines and walls. The automated solver must find a complete gem-collection sequence with minimal moves.

Three algorithms are examined in depth: Breadth-First Search (BFS), which guarantees the shortest-move solution; Greedy Best-First Search, which employs a heuristic-driven priority queue for faster, near-optimal solutions; and Divide & Conquer (D&C), which applies Merge Sort’s recursive halving strategy combined with Quick Sort’s spatial ordering to eliminate the exponential state space.

Through formal Big-O derivations, worked numerical examples, visual comparisons, and live benchmark data, this report demonstrates that D&C achieves a dramatic reduction from O(2G) exponential complexity to O(G2) polynomial complexity, trading a 10–20 % path-length overhead for up to 100× faster computation on large gem counts.

Problem Statement & Game Mechanics

2. Problem Statement & Game Mechanics

2.1 The Inertia Puzzle

Inertia is a grid-based puzzle where the player occupies a cell on an R × C grid. When the player moves in one of eight directions (N, NE, E, SE, S, SW, W, NW), they slide continuously in that direction until a wall, stop tile, or the grid boundary halts their motion. The objective is to collect every gem on the board while avoiding mines.

2.2 Key Entities

Entity Symbol Effect on Player
Gem 💎 Collected on contact; must collect ALL to win
Mine 💣 Instant game-over on contact
Wall / Stop 🧱 Halts the slide; player stops adjacent
Open Cell ⬜ Player slides through freely
Boundary — Stops the slide at the edge of grid

2.3 The Slide Simulation Cost

In most graph problems, moving to a neighbour costs O(1). In Inertia, determining the destination of a slide requires simulating every intermediate cell:

Slide Simulation Cost → L = max(Rows, Cols)
To find where one directional move ends, we step through up to L cells. This is the atomic unit of work for all three algorithms.

2.4 Formal Problem Definition

Let the grid be R × C, containing G gems at fixed positions {g₁, g₂, …, gG}. The solver must produce a sequence of directional moves M = [m₁, m₂, …, mₖ] such that executing M from the start position visits every gem position and avoids all mines. The quality metrics are:

  • Completeness – Does the algorithm always find a solution if one exists?
  • Optimality – Does it find the solution with the fewest moves?
  • Time Complexity – How does runtime scale with grid size and gem count?
  • Space Complexity – How much memory does the search require?

State Space Formulation

3. State Space Formulation

3.1 Defining a State

For any search algorithm to solve Inertia, it must track more than just the player's current position. A complete State is the pair:

State = (Position, CollectedGems)
 • Position: (x, y) — one of R × C cells
 • CollectedGems: a subset of {g₁…gG} — one of 2G possible subsets

3.2 State Space Size

Combining both components gives the total number of distinct states the search may visit: 

Total States (V) = Rows × Cols × 2^G

Example (10×15 grid, 10 gems):
  V = 10 × 15 × 2^10
    = 150 × 1024
    = 153,600 states

3.3 Why 2G is the Bottleneck

With G = 10 gems, the power-set of collected gems has 1 024 entries. With G = 15, it becomes 32 768 entries. This exponential growth is the central algorithmic challenge the Divide & Conquer approach was designed to defeat.

G (Gems) 2G States G2 (D&C) Ratio (Improvement)
5 32 25 1.3×
10 1,024 100 10.2×
15 32,768 225 145×
20 1,048,576 400 2,621×

3.4 Step-by-Step Complexity Calculation Method

When computing the complexity of any algorithm in this project, follow these four steps:

  1. Identify Input Sizes: N = R × C (grid cells), G = number of gems
  2. Identify the Basic Operation: The slide simulation at cost O(L)
  3. Count States: Ask "What makes two situations distinct?" → (position + gems held)
  4. Multiply: Total Complexity = (States) × (Work per State)
Master Formula:   Time = (R × C × 2G) × (8 directions × L)  →  Simplified: O(2G · R · C · L)

Algorithm I – Breadth-First Search

4. Algorithm I – Breadth-First Search (BFS)

BFS — solveGame()
Time: O(2G·R·C·L)  Space: O(2G·R·C)

Goal: Find the minimum-move solution by exploring states layer-by-layer.

4.1 How BFS Works in Inertia

BFS maintains a FIFO queue of states. At each iteration it dequeues the "shallowest" state, tries all 8 slide directions, and enqueues any new, unvisited states. The first time all gems appear in a state's collected set, the solution path is returned.

4.2 Complexity Derivation

Component Cost Reason
Total unique states R × C × 2G Position × gem-subset combinations
Directions tried per state 8 Fixed 8-directional grid
Slide simulation per direction O(L) = O(max(R,C)) Step through cells until stopped
Queue enqueue / dequeue O(1) amortised Array-based FIFO
Total Time O(R·C·2G·L) States × work per state
Total Space O(R·C·2G) Visited set + queue

4.3 Annotated Code Complexity

async function solveGame() {
    let queue = [];                          // Init: O(1)
    queue.push({ x, y, collected:[], path:[] });

    while (queue.length > 0) {              // Outer: O(States) = O(R·C·2^G)
        let current = queue.shift();        // Dequeue: O(1) amortised

        if (current.collected.length === allGems.length) return; // Win check: O(1)

        for (let dir of directions) {       // 8 directions: O(8) = O(1)
            // Slide simulation:            O(L)
            while (true) { /* step-by-step slide */ }

            queue.push({...});             // Enqueue: O(1)
        }
    }
}
// TOTAL: O(R·C·2^G) × O(8×L) = O(R·C·2^G·L)

Algorithm II – Greedy Best-First Search

5. Algorithm II – Greedy Best-First Search

Greedy — solveGameSorted()
Time: O(V² log V)  Space: O(V)

Goal: Find a solution faster than BFS by always expanding the most promising state first, guided by a heuristic score.

5.1 The Heuristic Function

At each state, a score is computed to rank how "good" the state is:

Score = (Gems Remaining × 1000) + Distance to Closest Remaining Gem

Lower score → state is preferred (fewer gems left, closer to next gem).
The queue is sorted ascending by score at every iteration, simulating a priority queue.

5.2 Complexity Derivation

Component Cost Reason
Loop iterations (states V) O(V) = O(R·C·2G) Same state space as BFS
Slide simulation per direction O(L) Same as BFS
Heuristic calculation per state O(G) Scan all remaining gems for min distance
Queue re-sort per iteration O(Q log Q) Q ≈ V in worst case → O(V log V)
Total Time (worst case) O(V² log V) V iterations × V log V sort
Total Time (ideal heap) O(V log V) With a proper binary heap
Space O(V) Queue + visited, same as BFS

5.3 Why Greedy is Faster in Practice

Despite the higher worst-case bound, Greedy typically explores far fewer states than BFS because the heuristic drastically narrows the effective search frontier. The algorithm behaves like a laser — focusing exclusively on paths toward gems rather than flooding the entire state space.

5.4 Sorting Algorithm Used: Quick Sort

JavaScript's Array.prototype.sort() uses Timsort (a hybrid of Merge Sort and Insertion Sort), but conceptually behaves like Quick Sort for average-case O(Q log Q) performance. The sort is called on the queue after every state expansion, adding a log factor to the per-iteration cost.

5.5 Properties

Property Greedy Value
Complete? ✅ Yes
Optimal (fewest moves)? ❌ No — heuristic can follow suboptimal paths
Practical speed ⭐ Fastest in practice
Sorting overhead O(Q log Q) per iteration via Quick Sort/Timsort
Greedy Trap (known failure case): If a gem sits behind a U-shaped wall, the heuristic drives the solver toward the wall repeatedly. BFS naturally flows around it because it has no directional bias. This demonstrates a key limitation of greedy heuristics — they can become trapped in local optima.

Algorithm III – Divide & Conquer

6. Algorithm III – Divide & Conquer (D&C)

D&C — solveGameDNC()
Time: O(G²·R·C·L) + O(G log G)  Space: O(G·d)

Goal: Eliminate the exponential 2G term by recursively splitting gems into spatial clusters — exactly mirroring Merge Sort's divide-and-conquer pattern.

6.1 Algorithm Steps

Step 1 — Divide (Sort & Split using Quick Sort + Merge Sort Pattern)

Sort all gems by their X or Y bounding-box axis using Array.sort() (Quick Sort). Then split the sorted list at the midpoint into Group A and Group B — exactly as Merge Sort splits its array in half.

Step 2 — Conquer (Recursion)

Recursively call solveGameDNC(gems) on each half. The base case arises when a subgroup has exactly one gem: a simplified BFS is run that ignores the gem-collection state (no 2G complexity), costing only O(R·C·L).

Step 3 — Combine (Merge)

Connect the solution paths: try both orderings (A → B and B → A) and keep the shorter total path. This is analogous to Merge Sort's merge step that combines two sorted halves.

6.2 D&C Recursion Tree (for 6 gems)

         [G1, G2, G3, G4, G5, G6]          ← Quick Sort by X, then split
               /             \
      [G1, G2, G3]        [G4, G5, G6]     ← Quick Sort again, split
        /      \              /      \
   [G1,G2]   [G3]       [G4,G5]   [G6]
    /   \                 /   \
  [G1] [G2]           [G4] [G5]

  Each leaf → simple BFS (no 2^G overhead)
  Each merge → try A→B and B→A, keep shorter

6.3 Complexity Derivation

Component Cost Source
Sort gems at each level O(G log G) Quick Sort via JS Array.sort()
Recursion depth O(log G) Halving like Merge Sort
BFS base case (per leaf) O(R·C·L) Simple single-gem BFS
Total subproblems O(G) One leaf per gem
Try both orderings 2× per merge Doubles work → G² growth
Total Time O(G²·R·C·L) + O(G log G) Polynomial, not exponential
Space O(G·d) Recursion stack depth × path length

6.4 The Key Breakthrough

BFS: 210 = 1 024 gem-state multiplier for 10 gems
D&C: 102 = 100 gem-state multiplier for 10 gems
→ 10.2× fewer states explored; trade-off is ~10–20% longer output path.

7.4 The Visited Set Optimisation (BFS)

Without a visited set, BFS explores up to 8d paths (approximately 35 trillion for d=15). The visited set prunes revisited states, collapsing the search from exponential-in-depth to polynomial-in-grid-size exponential-in-gems.

Without pruning: O(815) ≈ 35 trillion states  With visited set: O(R·C·2G) ≈ 153 thousand states
→ The visited set is the most critical optimisation in BFS.

7.5 Why Sorting Adds the Log Factor (Greedy)

Each time Greedy re-sorts the queue of Q states using Quick Sort, it incurs O(Q log Q) overhead. Over V iterations with Q growing toward V, this accumulates to O(V² log V) in the worst case. A proper binary heap (min-heap priority queue) would reduce this to O(V log V).

Benchmarks were run on the Inertia Game solver in a browser environment (Chrome V8 engine). Each test configuration was run five times; the median value is reported. Grid sizes and gem counts were varied systematically.

8.3 Key Observations

Observation 1 — BFS dominates small instances: For ≤8 gems, BFS runs within 1 second and guarantees the optimal path. Greedy is slightly faster but longer.
Observation 2 — Greedy wins the sweet spot: For 8–12 gems, Greedy is consistently 2–6× faster than BFS with only 1–2 extra moves in the solution.
Observation 3 — D&C scales to large inputs: At 12 gems, D&C was 100× faster than BFS (0.1 s vs. 10 s), confirming the polynomial vs. exponential gap. Path quality degrades by ~15–20% on average.
Observation 4 — Greedy fails on "trap" layouts: U-shaped wall formations cause Greedy to waste cycles near the wall. BFS always escapes such local optima.

Conclusions & Recommendations

9. Conclusions & Recommendations

9.1 Summary of Findings

This case study has demonstrated that no single algorithm dominates across all game configurations. Each algorithm occupies a distinct position in the optimality–speed–scalability triangle:

Algorithm Strengths Weaknesses Ideal Use
BFS Optimal path; always completes; no heuristic bias Exponential memory and time as G grows Small grids (G ≤ 8), correctness-critical
Greedy Fastest in practice; good path quality Not optimal; can trap on local optima Medium grids, interactive play, speed demos
D&C Polynomial scaling; educates on sorting; low memory ~15% path overhead; slower than Greedy at small G Large grids (G ≥ 10), educational demonstrations

9.2 Key Theoretical Takeaways

  1. All algorithms face exponential worst-case from gem-collection state space — the 2G factor is fundamental to the problem structure.
  2. BFS guarantees shortest path by exploring nodes uniformly; the visited set is its most critical optimisation, reducing states from trillions to thousands.
  3. Greedy adds a log factor through queue sorting (Quick Sort / Timsort), but pays off in dramatically reduced states explored.
  4. D&C transforms the problem from Exponential O(2G) to Polynomial O(G²) by applying Merge Sort's recursive halving and Quick Sort's spatial ordering — a fundamental demonstration of how sorting algorithms enable algorithmic breakthroughs in unrelated domains.
  5. The visited set is the single most impactful micro-optimisation across all three algorithms.

9.3 Recommendations for Future Work

R1: Replace Greedy's linear re-sort with a binary min-heap (JavaScript PriorityQueue), reducing time from O(V² log V) to O(V log V).
R2: Explore A* Search — combining BFS's optimality guarantee with Greedy's heuristic to achieve both speed and shortest-path correctness.
R3: Extend D&C to consider k-d tree spatial partitioning for even smarter gem cluster splitting, potentially reducing G² to G log G.
